## Issue 1: Add Python SDK client

**Label:** `good first issue`, `sdk`  
**Difficulty:** Medium  
**Skills:** Python

### Description

ClearAgent has a TypeScript SDK stub (`packages/sdk`) but no Python client. Many compliance teams use Python for data analysis and regulatory reporting. A Python client would make ClearAgent accessible to a much wider audience.

### Acceptance criteria

- `pip install clearagent` (or installable via `pip install .`)
- `ClearAgentClient(api_key, base_url)` constructor
- `.verify(input_data)` ‚Üí returns `(job_id, status)`
- `.get_job(job_id)` ‚Üí returns job status and `content_hash`
- `.get_events(filters)` ‚Üí returns paginated event list
- `.submit_review(event_id, action, justification, reviewer)` ‚Üí returns review ID
- `.get_integrity()` ‚Üí returns `valid_chain`, `merkle_root`
- Type hints throughout
- README in the package directory with install + usage example

### Getting started

Look at the TypeScript SDK stub in `packages/sdk/src/index.ts` for the intended interface shape. The [API reference](api-reference.md) documents all endpoints and request/response schemas.

## Issue 2: Add X-Request-ID to all API responses

**Label:** `good first issue`, `api`  
**Difficulty:** Easy  
**Skills:** TypeScript, Express

### Description

The API generates a `requestId` for each request (see `packages/api/src/index.ts` middleware) but does not include it in the response headers. Adding `X-Request-ID` to all responses enables client-side correlation of requests with server logs.

### Acceptance criteria

- All API responses include an `X-Request-ID` header
- The value matches the `requestId` generated by the existing middleware
- If the client sends `X-Request-ID` in the request, that value is echoed back (not a new UUID)
- TypeScript and lint pass: `npm run typecheck && npm run lint`

### Getting started

The request ID middleware is in `packages/api/src/index.ts`. The response middleware should be added nearby. Look at how other Express middleware adds headers (e.g., `helmet()`) for the pattern.

## Issue 3: Add curl examples to API reference

**Label:** `good first issue`, `documentation`  
**Difficulty:** Easy  
**Skills:** Documentation, curl, JSON

### Description

`docs/api-reference.md` has endpoint descriptions but the curl examples are basic. Some endpoints (especially `GET /v1/events` with filters, `GET /v1/audit/export` with authority params, and `GET /v1/events/stream`) would benefit from richer examples showing all query parameters and realistic payloads.

### Acceptance criteria

- Every endpoint has at least one working curl example
- The `GET /v1/events` example shows all query parameters (agent_id, status, decision, from, to, cursor, limit)
- The `GET /v1/audit/export` example shows the authority_name and authority_ref parameters
- The SSE endpoint example includes both curl and a complete JavaScript EventSource snippet
- All examples use the demo API key and localhost URLs
- Examples are tested against a running local instance

### Getting started

Run the [Quick Start](self-hosting.md#quick-start) to get a local instance running. Then test each curl command against it and capture the real response output to include as example responses.

## Issue 4: Add confidence score histogram to dashboard

**Label:** `good first issue`, `frontend`  
**Difficulty:** Medium  
**Skills:** TypeScript, React, data visualization

### Description

The dashboard's live feed shows individual events but there's no aggregate view of confidence score distribution. A histogram showing how many events fall into each confidence bucket (0‚Äì0.5, 0.5‚Äì0.7, 0.7‚Äì0.85, 0.85‚Äì0.95, 0.95‚Äì1.0) would help compliance teams quickly spot patterns in agent confidence.

### Acceptance criteria

- New component `ConfidenceHistogram` in `packages/dashboard/src/components/`
- Rendered on the main dashboard page alongside (not replacing) the live feed
- Buckets: `[0, 0.5)`, `[0.5, 0.7)`, `[0.7, 0.85)`, `[0.85, 0.95)`, `[0.95, 1.0]`
- Updates in real-time as new events arrive via SSE
- Uses only the existing Tailwind CSS ‚Äî no additional chart library dependencies
- Responsive: works at both mobile and desktop widths

### Getting started

Look at `packages/dashboard/src/pages/LiveFeed.tsx` for how events are fetched and how SSE is consumed. The `GET /v1/events` endpoint returns `confidence` (0.0‚Äì1.0) on each event. Start with static data from the initial fetch before adding real-time updates.

## Issue 5: Write integration test for hash chain integrity

**Label:** `good first issue`, `testing`, `compliance`  
**Difficulty:** Medium  
**Skills:** TypeScript, PostgreSQL, testing

### Description

The hash chain is ClearAgent's core compliance guarantee. It should have a dedicated integration test that proves the integrity endpoint detects tampering. This is the most important test in the codebase from a compliance perspective.

### Acceptance criteria

Test file at `packages/api/src/__tests__/hash-chain-integrity.test.ts` covering:

1. **Happy path** ‚Äî Seed 5 events ‚Üí `GET /v1/audit/integrity` ‚Üí `validChain: true`
2. **Broken chain detection** ‚Äî Seed 5 events ‚Üí use raw SQL to corrupt one event's `content_hash` directly in the database (bypassing the application layer) ‚Üí `GET /v1/audit/integrity` ‚Üí `validChain: false`, `brokenAt: <eventId>`
3. **Empty chain** ‚Äî Fresh database ‚Üí `GET /v1/audit/integrity` ‚Üí `validChain: true`, `totalEvents: 0`
4. **Single event** ‚Äî One event ‚Üí `validChain: true`, `totalEvents: 1`

**Important:** Step 2 must use a raw database connection to modify `content_hash` directly, since the application enforces immutability. This is intentional ‚Äî it tests that the integrity check catches database-level tampering that bypasses application controls.

### Getting started

Look at `packages/api/src/services/hashChain.ts` for the hashing logic and `packages/api/src/routes/audit.ts` for the integrity endpoint. The test requires a real PostgreSQL instance ‚Äî see the existing test setup for how to configure the test database connection.
